# Issue原稿作成スキル

## 概要

チャットでユーザーが伝えた概要クラスの情報をもとに、`gh-issue`スキルへ渡すためのIssue原稿を作成し、`.claude/tmp/issue/`に保存する。

## ワークフロー

### Step 1: 実行原則の確認

Skillツールで `my-rfp` を実行し、実行原則を確認する。

### Step 2: PLANモードに入る

EnterPlanModeツールを使用してPLANモードに入る。

### Step 3: 対象の探索

Taskツール（subagent_type=Explore）で関連仕様書を探索する。

探索の起点：
- `_llm-docs/project.md`（プロジェクト全体の仕様エントリーポイント）

上記を起点に、ユーザーの要件に関連する仕様書パスを特定する。
ユーザーが参考リンク・資料を提示している場合はあわせて収集する。

### Step 4: 原稿の作成（会話内）

以下のテンプレートで原稿を作成する（**この時点ではファイルに保存しない**）。

```markdown
## 概要

{ユーザーの説明から要約}

### 資料

{仕様書以外の参考ドキュメントリンク。なければこのセクションごと省略}

## やること

- [ ] {タスク1}
- [ ] {タスク2}

## ここではやらないこと

{ユーザーがスコープ外を明示した場合のみ記載。なければこのセクションごと省略}

- {スコープ外の内容}

## 確認観点

- [ ] {テスト・動作確認・レビュー観点1}
- [ ] {テスト・動作確認・レビュー観点2}

## 仕様書

- {Step 3 で特定した関連仕様書パス1}
- {Step 3 で特定した関連仕様書パス2}
```

**作成ルール**：
- 設計方針としてYAGNI（今必要でないものは作らない）とDRY（重複を避ける）に従う
- AIが一般的なベストプラクティスや想定要件を独自に追加することは禁止
- 不足している情報は独自判断で補完せず、ユーザーに確認する
- 以下の情報源のみをもとに作成する（my-rfp原則に従う）：
  - ユーザーがチャットで伝えた言葉
  - ユーザーがチャットで伝えた資料
  - `_llm-docs`配下の仕様書
  - 既存コードのパターン
  - 明示された技術スタックの公式リファレンス
- `## 仕様書`にはStep 3で確認した`_llm-docs/`配下の関連仕様書パスのみ記載する

### Step 5: Codex CLIレビュー（最大10回）

Skillツールで`doc-review`を実行し、作成した原稿をレビューする。
引数として原稿の内容または保存パスを渡す。

> **注意**: Codex CLIがインストールされていない場合はこのステップをスキップし、Step 6に進む。

指摘事項があれば原稿を修正し、最大10回まで繰り返す。

### Step 6: ファイルへの保存

ExitPlanModeツールで計画の承認を得た後、`.claude/plans/issue/[slug]-[timestamp].md` に保存する。

- `[slug]`: 内容を表す英語のケバブケース（例: `add-search-feature`, `fix-login-flow`）
- `[timestamp]`: `YYYYMMDDHHmmss` 形式

### Step 7: PLANモードを終了・次のアクションを案内

保存後、以下をユーザーに提示する：

```
保存しました: .claude/tmp/issue/[slug]-[timestamp].md

Issue を作成するには:
/gh-issue .claude/tmp/issue/[slug]-[timestamp].md
```
